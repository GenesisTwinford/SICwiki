# 要件定義（実装向け整理版）

## 1. 目的・コンセプト

- 本プロダクトは、プログラミングなど理系分野の複雑な学問について、初学者・独学者が「全体構造」と「現在地」を把握しながら学べるコース型学習サービスである。
- 教材制作はWiki的に行い、複数人の有志が作成・改善できるようにする。
- 学習継続はDuolingo的に支援し、進捗・競争・達成感を通じて継続率を高める。
- 理解深化はブログ的に支援し、文章での整理に加えてコメント/補足機能で差別化する。

## 2. スコープ

### 2.1 MVP（今回実装対象）

- ホーム画面（`/`）
- 記事閲覧・評価（`/article/:articleId`）
- 記事編集（`/article/:articleId/edit`）
- 記事作成（`/article/new`）
- マイページ（`/mypage/:userId`）
- 認証（ログイン/ログアウト）
- フォロー関係データの保持（UIなし）

### 2.2 将来実装（MVP外）

- 編集履歴閲覧
- 採用記事以外の記事閲覧
- 問題回答状況の自動保存
- 活動量スコアリング拡張（テストクリア、説明追加、誤字修正、コメント評価）
- バッジ、ストリーク、レベル制度
- 記事難易度設定
- フォロー/フォロワー管理UI
- 通知表示（`/` 左エリア）
- 設定画面（`/setting`：通知設定、支払い情報）

## 3. 用語定義

- コース: 学習対象を体系化した単位。木構造・入れ子構造で記事へ到達できる。
- 記事: タイトル・概要・複数のセクション・問題集で構成される学習コンテンツ。
- セクション: 記事内の分割単位。編集・評価・補足の対象となる。
- 問題集: セクションに紐づく問題。MVPではコード並べ替え方式を基本とする。
- 活動量: ユーザーの学習・貢献に基づく指標。MVPでは日次進捗と、保持済みのフォロー関係に基づくフレンド内順位を扱う。

## 4. 権限・アクセス制御

- 記事閲覧（`/article/:articleId`）のみログイン不要。
- 上記以外の画面・操作は、ルートを含めログイン必須。
- 作成/編集/コメント/リアクション/プロフィール編集は認証済みユーザーのみ。
- 記事編集権限は「ログインユーザー全員」に付与する。
- マイページ（`/mypage/:userId`）の `userId` は公開用slugを指す。MVPでは本人のページのみ利用対象とする。

## 5. 機能要件（MVP）

### FR-001 認証

- ログイン機能を提供する。
- ログアウト機能を提供する。
- 未ログインで保護ページにアクセスした場合、ログイン画面またはログイン導線へ遷移させる。

受け入れ条件:

- 認証済み状態で保護ページへ遷移できる。
- ログアウト後、保護ページは再アクセス時に認証が必要となる。

### FR-010 ホーム画面（`/`）

- 3カラム構成を提供する。
- 左カラムにロゴとプロフィール情報を表示する。
- 中央カラムにコース一覧を表示する。
- コース一覧は木構造または入れ子構造で、目的の記事へ辿れるUIにする。
- 右カラムにユーザー統計を表示する。
- ユーザー統計には、デイリータスク進捗とフレンド内での活動量順位を含める。

受け入れ条件:

- コースの階層を展開/折りたたみし、目的記事へ遷移できる。
- デイリータスクの進捗がログインユーザーに応じて表示される。

### FR-020 記事閲覧（`/article/:articleId`）

- 記事をログイン不要で閲覧できる。
- 記事は「タイトル -> 概要 -> セクション -> 問題集 -> セクション -> 問題集 ...」の順で表示する。
- 各セクション/問題集はバージョン管理対象とする。
- 編集ボタンを表示する。認証済みユーザーが押下した場合は `/article/:articleId/edit` へ遷移し、未認証ユーザーにはログイン導線を表示する。

受け入れ条件:

- 未ログインでも記事詳細ページが表示される。
- 表示順が定義どおりである。
- 認証済みユーザーは編集ページに遷移でき、未認証ユーザーにはログイン導線が表示される。

### FR-021 記事作成

- 認証済みユーザーは記事を作成できる。
- 作成時にタイトル、概要、1件以上のセクションを登録できる。
- セクションに問題集（コード並べ替え方式）を紐づけられる。

受け入れ条件:

- 必須項目入力で記事が作成され、記事詳細へ遷移する。
- 作成直後の記事は閲覧可能である。

### FR-030 記事編集（`/article/:articleId/edit`）

- セクション単位で記事を編集できる。
- 編集結果は新しいバージョンとして保存できる。

受け入れ条件:

- 任意セクションの本文変更が可能。
- 保存後、該当セクションのバージョンが増加する。

### FR-040 記事評価（セクション単位）

- 各セクションに対して、以下の3軸で評価できる。
- 上級者向け <-> 初心者向け
- わかりやすい <-> わかりにくい
- 正確 <-> 不正確
- 評価結果を記事採用判定の材料として保持する。
- 採用判定の母集団は、同一コースに属する記事候補群とする。
- 評価送信時または現行版更新時に、同一コースの記事候補群に対して採用スコアをアプリケーション側で再計算し、採用状態を保存する。
- MVPの通常導線では採用記事のみを表示する。採用記事以外の個別閲覧UIは将来実装とする。
- 採用記事判定は「わかりやすさ評価」と「レビュー数」を主指標とし、以下で算出する。
- 採用スコア = 0.7 × わかりやすさ平均スコア + 0.3 × レビュー数正規化スコア。
- レビュー数正規化スコアは同一記事候補群の最小値0、最大値1に線形正規化する。
- 採用候補の最低レビュー数は3件とし、3件未満は暫定表示（採用判定対象外）とする。
- 同点時は、正確性スコアが高いものを優先し、それでも同点なら更新日時が新しいものを優先する。

受け入れ条件:

- 各軸で評価を送信できる。
- 評価結果が保存され、採用判定ロジックが参照可能である。

### FR-050 セクション内補足・リアクション

- セクション内の文章単位で、以下を扱える。
- 自身の補足コメント
- 他ユーザーによる補足コメント
- 挫折箇所へのリアクション
- コメントとリアクションは、投稿対象のセクションバージョンにのみ紐づける。新しいバージョンへは自動で引き継がない。
- コメント削除は「自分のコメントのみ」可能とし、コメント表示時のメニューから削除できる。
- コメント/リアクションの通報機能はMVP対象外とし、将来機能として専用URLを追加する。

受け入れ条件:

- 文章単位にコメントを紐づけて投稿/表示できる。
- リアクション数または状態が表示される。

### FR-060 マイページ（`/mypage/:userId`）

- 自身のプロフィールを閲覧できる。
- 自身のプロフィールを編集できる。
- 自身の学習統計を表示できる。
- 他ユーザーのマイページ閲覧はMVP対象外とする。
- `userId` は公開用slugであり、内部のユーザー主キーとは分離する。

受け入れ条件:

- ログインユーザーが自分のプロフィール情報を更新できる。
- 学習統計が表示される。

### FR-070 デイリータスクと順位集計

- デイリータスクは固定方式とし、1日1タスクを配信する。
- 問題を3問以上解くことを1タスクとする。問題の種類は問わない。
- デイリー進捗は未達成/達成の2値で管理する。
- フォロー関係データはMVPで保持するが、作成・削除・一覧表示のUIは対象外とする。
- フレンド内活動量順位は週間集計（JST、月曜00:00開始、日曜23:59終了）で更新する。
- 活動量の週間スコアは以下で算出する。
- デイリータスク達成: 1日あたり10点
- 問題集の採点実行: 1問正解につき1点

受け入れ条件:

- 日次で達成状態が記録される。
- 週間ランキングがフレンド（片方向フォロー先）集合で表示される。

## 6. データ要件（MVPで最低限必要な概念）

- User
- Profile
- Course（階層構造を持つ）
- Article
- CourseArticleAdoption
- ArticleSection
- SectionVersion
- Quiz（コード並べ替え）
- QuizVersion
- SectionRating（3軸評価）
- AnnotationComment（文章単位コメント）
- FrustrationReaction
- QuizSubmission
- DailyTaskProgress
- FriendActivityRank
- FollowRelation（片方向フォロー）

確定仕様:

- フレンド関係は片方向フォローとする。
- 採用記事は FR-040 の採用スコアで決定する。
- 問題集は「ブロックのドラッグアンドドロップによる並べ替え」と「文字入力による正誤判定」を併用し、採点タイミングは採点ボタン押下時とする。
- コメント/リアクションはセクションバージョンをまたいで自動継承しない。

## 7. API/画面設計時の実装前提

- フロントエンド: Next.js、shadcn/ui、Tailwind CSS
- バックエンド: Hono
- DB: Turso
- ORM: Drizzle
- 認証: better-auth
- デプロイ: Cloudflare

実装前提ルール:

- ルートごとに認証要否を明示する。
- 更新系処理は監査可能な形で更新者/更新時刻を保持する。
- バージョン管理対象（セクション/問題集）は履歴を破壊しない。
- ユーザー内部IDはランダムID（推奨: ULID）を使用し、`userId` は公開用の一意なslugを使用する。
- 採用記事判定は同期API内または非同期ジョブ内のいずれかで再計算してよいが、計算責務はアプリケーション側に置く。
- 記事公開範囲はMVP時点で全公開のみとする。

## 8. 非機能要件（実装可能な形で定義）

### NFR-001 パフォーマンス

- 主要画面（`/`, `/article/:articleId`, `/mypage/:userId`）で以下を満たす。
- 初回表示時のLCP: 2.5秒以下（ローカル開発相当の標準的なネットワーク条件）。
- CLS: 0.1以下。
- INP: 200ms以下。
- 計測は Chrome DevTools / Lighthouse を使用する。

### NFR-002 セキュリティ

- 認証必須ページは未認証アクセス時に保護される。
- 主要入力（記事本文、コメント、プロフィール）でXSS対策を行う。
- 更新系APIはCSRF対策または同等の保護策を行う。
- 認可チェック（本人のみ編集可など）をサーバー側で必須化する。

### NFR-003 可用性・運用

- エラー時にユーザー向けメッセージを表示し、致命的なクラッシュを避ける。
- サーバー/APIエラーを追跡可能なログを残す（request id, user id, endpoint, status）。

### NFR-004 保守性

- APIルート・DBスキーマ・画面コンポーネントに責務分離を行う。
- ESLintエラー0件を維持してマージする。
- 主要機能に対して最低限の結合テストまたはE2Eテストを追加する。

## 9. 要確認事項（不明点の明記）

- 記事公開範囲は現在「全公開のみ」だが、将来変更（下書き/限定公開追加）の可能性がある。
- 問題UIの詳細挙動（ドラッグ操作のアクセシビリティ要件、モバイル最適化、キーボード操作定義）は画面設計時に確定する。
